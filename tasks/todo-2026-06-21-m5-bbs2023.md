### Verify dispatch (uniform — no BBS-specific caller code)
`DataIntegrityProofPipeline.VerifyAsync(JsonElement, IVerificationMethodResolver, options, ct)` resolves the proof's `verificationMethod` → `ResolvedVerificationMethod.PublicKey` (`PublicKeyMaterial`), dispatches by `proof.Cryptosuite` name (`GetByName` → fallback `GetByProofType`), and calls `suite.VerifyProof(reveal, proof, publicKey)`. **A bbs-2023 derived proof on a reveal document verifies through the existing `DataIntegrityMechanism` with no change**, given the suite is registered.

### Keys
`PublicKeyMaterial.FromMultikey(string)` yields a `KeyType.Bls12381G2` key for the BLS multikey (multicodec `0xeb05`, 96-byte compressed G2 point); the existing `NetDidVerificationMethodResolver` already produces it. `NetCrypto.DefaultKeyGenerator.Generate(KeyType.Bls12381G2)` mints a BLS keypair (raw `PrivateKey` + `MultibasePublicKey`) — used by the dataproofs BBS sample, so tests can mint BLS keys + a `did:key`.

### `NetCrypto.IBbsCryptoProvider`
`bool IsAvailable`; `byte[] Sign(privKey, IReadOnlyList<byte[]> messages, header=default)`; `byte[] DeriveProof(pubKey, signature, messages, revealedIndices, presentationHeader, header=default)` (**no private key**); `byte[]…VerifyProof(...)`. `BbsUnavailableException` on a host without the native lib. Confined inside `Bbs2023Cryptosuite` — credentials-dotnet never touches it directly.

## 2. Scope decision (resolving the F2 gate + the holder boundary)

bbs-2023 has a three-stage lifecycle (base-issue → derive → verify). The recon fixes which stages M5 can ship:

- **VERIFY (FR-042) — ships, free.** Registering `Bbs2023Cryptosuite` as an `ICryptosuite` makes a derived bbs-2023 proof verify through the **existing** `DataIntegrityMechanism` + `DefaultVerifier` (uniform dispatch + BLS key resolution + `BindIssuer` all already work). Zero verifier code.
- **DERIVE (FR-031) — ships.** `DeriveProof` is key-free and public. M5 exposes a focused public **`IBbsDeriver`** in `Credentials.Rdfc` that wraps it, drawing the presentation header from the engine RNG seam (`IRandomSource`, F9). This is the holder selective-disclosure operation, delivered minimally — **not** the full `IHolder` (which, per the M4 precedent, stays in M6 and can later absorb `IBbsDeriver`).
- **ISSUE (FR-014) — GATED, not shipped.** There is **no** ISigner/IKeyStore BBS base-proof API; the only path takes a **raw private key**, and shipping that would make credentials-dotnet export a private key (violating FR-015/NFR-006) or hand-attach a proof (violating FR-050). Per the master plan's F2 decision, **BBS issuance is gated until DataProofs ships a key-store BBS create API.** A `bbs-2023` issuance attempt through the standard path fails fast (the substrate's `NotSupportedException`); M5 surfaces a clear credentials-dotnet gate message and skip-marks the issuance tests. **`Bbs2023BaseRequest` is deferred until F2 lands** (a public request type that can't be fulfilled would be dead surface).
- **`InspectBbsBase` / `BbsDisclosureMap` — deferred.** Extracting "which pointers are mandatory" from a base proof needs the base-proof CBOR metadata (`Bbs2023ProofValue.ParseBaseProof` is `internal` to DataProofs — no public API). The deriver takes caller-supplied reveal pointers; a holder reads the base credential's claims through the normal `Credential` accessors. Deferred until DataProofs exposes base-proof metadata (or M6).

**Net M5 deliverable:** end-to-end **derive → verify** for bbs-2023 selective disclosure (unlinkable, mandatory-group-enforced), with issuance correctly gated. This is coherent and testable for real (the BLS native lib is in the closure via NetCrypto), with base credentials crafted via the substrate's raw-key API **in tests only** (the M4 `CraftSdJwt` precedent).

## 3. Design decisions

1. **bbs-2023 is a Data Integrity cryptosuite — no new `SecuringForm`/`SecuringState`.** A derived bbs-2023 proof is an embedded `proof` with `cryptosuite=bbs-2023`; the derived/reveal credential is `SecuringState.DataIntegrity` and verifies through the existing DataIntegrity proof path. No new form, no new verifier branch (FR-053: a suite by string).

2. **Register the suite in `Credentials.Rdfc` via a new `UseBbs2023()` builder method** (granular opt-in, alongside `UseRdfcSuites()`): `builder.Services.AddSingleton<ICryptosuite>(new Bbs2023Cryptosuite())` (idempotent marker, mirroring the existing RDFC registration) **plus** the deriver registration. The DI-collected `CryptosuiteRegistry` picks it up → verify works; `ISecuringCapabilities.AvailableDataIntegritySuites` now lists `bbs-2023`. Registration always succeeds even if the BBS native lib is absent (construction never throws); a runtime BBS-unavailable verify maps to **Indeterminate** (operational fault, fail-closed), a derive throws a clear capability error.

3. **`IBbsDeriver` (public, `Credentials.Rdfc`) is the holder derive seam — the sole caller of `Bbs2023Cryptosuite.DeriveProof` (FR-050).**
   ```csharp
   namespace Credentials.Rdfc;
   public interface IBbsDeriver
   {
       Task<Credential> DeriveAsync(Credential baseCredential, BbsDisclosureRequest request, CancellationToken ct = default);
   }
   public sealed record BbsDisclosureRequest
   {
       // RFC 6901 JSON Pointers to reveal IN ADDITION to the always-revealed mandatory group.
       public IReadOnlyList<string> RevealPointers { get; init; } = [];
   }
   ```
   The internal `Bbs2023Deriver` impl: `base = baseCredential.AsElement()`; `ph = _rng.GetBytes(32)` (F9, unlinkability); `reveal = Bbs2023Cryptosuite.DeriveProof(base, request.RevealPointers, ph)`; return `Credential.FromDocument(CredentialDocument.FromElement(reveal, ParsedElement), SecuringState.DataIntegrity)`. `IsAvailable`-gated (BBS absent → a clear `NotSupportedException`/`InvalidOperationException`, never a raw FFI fault). Maps `ProofGenerationException` (bad base/pointers) → `CredentialFormatException` (malformed input). **No draft type on `IBbsDeriver`'s surface** — `Bbs2023Cryptosuite` is confined to the impl (NFR-005).

4. **`Credentials.Rdfc` gains `InternalsVisibleTo` to use the internal RNG seam + `CredentialDocument.FromElement`.** The deriver needs `IRandomSource` (internal, `Credentials.Cryptography`) for F9. Add `Credentials.Rdfc` to `Credentials.Core.csproj`'s `InternalsVisibleTo` (it is a first-party opt-in extension package — the same trust level as the DI package, which is already listed). This avoids widening Core's public surface with the RNG seam.

5. **Issuer binding is inherited unchanged (M1).** A derived proof carries the issuer's `verificationMethod`; `DefaultVerifier.BindIssuer` requires `BaseDid(vm) == credential.Issuer.Id`. Therefore the **`issuer` member must be in the issuer's mandatory group** (so it is revealed and bindable) — a conformant bbs-2023 issuer makes `issuer`/`@context`/`type` (and any validity/status claims) mandatory. A reveal document that hides `issuer` fails binding (`issuer_binding_missing` ⇒ Failed) — correct. (M5 documents this; the verifier needs no change.)

6. **Mandatory disclosure is the substrate's cryptographic guarantee; the holder cannot hide a mandatory claim** (the recomputed `bbsHeader` would not match → verify fails). The holder can only hide claims the issuer marked selectively-disclosable. **Residual (inherent to selective disclosure, same class as M4):** if a *non-conformant* issuer marks a validity/status/`issuer` claim selectively-disclosable, a holder can hide it and the verifier cannot detect the omission. The defence is issuer-side (mandatory pointers); M5 documents it. Since M5 does **not** issue, there is no issuance guard to add here.

7. **F9 unlinkability:** the presentation header is fresh `IRandomSource.GetBytes(32)` per derive, so two derivations of the same base produce different derived proofs that both verify and are unlinkable — a tested DoD item.

## 5. Security model (carried invariants)

- **FR-050:** `Bbs2023Deriver` is the sole importer of `Bbs2023Cryptosuite.DeriveProof`; it never hand-builds/attaches/strips a `proof` (the substrate returns the assembled reveal document). Verify delegates entirely to the pipeline.
- **FR-015 / NFR-006 (the F2 reason):** credentials-dotnet **never** exports a raw BLS private key — which is exactly why issuance is gated, not shipped. The deriver is key-free.
- **NFR-005 / D3:** no `Bbs2023Cryptosuite` / dotNetRDF / Newtonsoft / CBOR type on any public surface — confined to `Bbs2023Deriver` + the registration. Extend the surface-confinement reflection test (in Rdfc.Tests, or assert the Rdfc public surface) to forbid `DataProofsDotnet.Rdfc.*` draft types.
- **FR-045 / F7:** a BBS-unavailable verify → **Indeterminate** (operational fault, never a crash, fail-closed); a bad derived proof → **Failed** (result-style `VerifyProof`); only malformed input propagates as `CredentialFormatException`.
- **Issuer binding (M1):** the derived proof's `verificationMethod` base DID must equal the (mandatory-revealed) `issuer` — to forge `issuer=victim` an attacker needs the victim's BLS key. Mandatory disclosure is cryptographically enforced; a hidden/altered mandatory claim fails verification.
- **NFR-002:** Core/DI default closure unchanged (System.Text.Json-only); BBS confined to the opt-in `Credentials.Rdfc` (which already carries the RDFC stack). Re-verified by `dotnet list … --include-transitive`.

## 6. Test plan (`tests/Credentials.Rdfc.Tests`)

House conventions: xUnit + FluentAssertions **7.0.0**, NSubstitute `.Returns(value)`. Container via `AddCredentials(b => b.UseNetDid().UseBbs2023())`. BLS keys via `DefaultKeyGenerator.Generate(KeyType.Bls12381G2)` + a `did:key`. Base credentials are crafted via `Bbs2023Cryptosuite.CreateBaseProofAsync` (raw key) **in a test helper only** (the engine never does raw-key issuance).

- **Derive → verify round trip (FR-031/042):** craft a base credential (issuer + subject with several properties; mandatory = `/issuer` + structural; selective = chosen subject props) → `IBbsDeriver.DeriveAsync(base, RevealPointers=[…])` → `VerifyCredentialAsync(derived)` ⇒ **Accepted**; proof/structure/validity `Passed`. Assert the derived document reveals the mandatory group + the selected pointers and **omits** the unselected ones.
- **Mandatory always revealed (DoD):** assert the derived reveal document contains the mandatory claims regardless of `RevealPointers`.
- **Unlinkability (F9 DoD):** derive twice from the same base with the same request ⇒ two **different** derived proofValues, **both** verify Accepted. (Confirms fresh RNG presentation headers.)
- **Mandatory tamper ⇒ Failed (security):** byte-edit a revealed mandatory claim in a derived document ⇒ `VerifyCredentialAsync` ⇒ **Rejected** (proof `Failed`) — the recomputed BBS header no longer matches.
- **Issuer binding (security):** a derived proof whose `verificationMethod` base DID ≠ the revealed `issuer` ⇒ **Rejected** (`issuer_binding`). A derive that hides `issuer` (if the base made it selective) ⇒ **Rejected** (`issuer_binding_missing`).
- **Tampered derived proof ⇒ Failed, not thrown (F7):** corrupt the derived `proofValue` ⇒ **Rejected** (`Failed`), never an exception.
- **Issuance gated (FR-014 DoD):** `issuer.IssueAsync(credential, new DataIntegrityIssuanceRequest { Cryptosuite = "bbs-2023", Signer, VerificationMethod })` ⇒ throws (the clear gate message); a `[Fact]` asserting the gate (not `Skip`, since the gate is a real, tested behaviour). A separate `Skip`-marked test documents the intended issue→derive→verify round trip for when F2 lands.
- **Capabilities / DI:** `AvailableDataIntegritySuites` contains `bbs-2023`; `IsSupported(SecuringSelector.DataIntegrity("bbs-2023"))` is true; `IBbsDeriver` resolves.
- **BBS-unavailable handling:** if `Bbs2023Cryptosuite.IsAvailable` is false on the host, the relevant tests are guarded/skip the crypto and assert the graceful path (Indeterminate verify / clear derive error). *(The BLS native lib is expected present via NetCrypto's closure; this guard is belt-and-suspenders.)*
- **NFR-005 surface confinement:** reflect over the `Credentials.Rdfc` public surface; assert no `DataProofsDotnet.Rdfc.*` / dotNetRDF / Newtonsoft / CBOR type appears.
- **NFR-002:** `dotnet list … --include-transitive` — Core/DI clean; Rdfc unchanged.

## 8. Risks & out-of-scope

- **Issuance (FR-014) is gated on DataProofs.** It ships the day DataProofs adds a BBS base-proof create API that takes an `IKeyStore` alias / `IBbsCryptoProvider` handle and returns the assembled document. Until then, `Bbs2023BaseRequest` + the issuer round trip are deferred (this is a **dependency/upstream** gap, not a credentials-dotnet defect — cross-repo hygiene). Flagged, not worked around.
- **`InspectBbsBase` / `BbsDisclosureMap` deferred** — needs public base-proof metadata (mandatory-pointer extraction) from DataProofs (`Bbs2023ProofValue` is internal).
- **Full `IHolder` is M6** — `IBbsDeriver` is the focused M5 derive seam; M6's `IHolder` can absorb it.
- **Selective-disclosure validity/status residual** (same class as M4): a non-conformant issuer that marks a validity/status/`issuer` claim selectively-disclosable lets a holder hide it; undetectable at the verifier (it looks like an absent claim). Defence is issuer-side (mandatory pointers); documented. M5 does not issue, so there is no issuance guard to add.
- **BBS native-lib availability** — `IsAvailable`-gated throughout; a host without the FFI degrades (verify → Indeterminate, derive → clear error), never crashes.
- **bbs-2023 is a W3C CRD draft (D3/NFR-005)** — confined behind `IBbsDeriver` + the suite registration; a re-pin is localized to DataProofs.

## 9. Review (2026-06-22)

**Status: complete and green.** Clean Release build (0 warnings, `TreatWarningsAsErrors` / XML-doc gate),
**275 tests pass** (135 Core + 119 DI + 21 Rdfc); 17 new M5 tests (incl. 7 adversarial regressions). The
Core/DI default closure is unchanged (NFR-002), and bbs-2023 is confined to the opt-in `Credentials.Rdfc`
package — **zero Core changes** (the deriver uses only the public `IRandomSource`/`Credential` surface; verify
is pure suite registration).

**Delivered (as planned, §3–§4):** `UseBbs2023()` (registers `Bbs2023Cryptosuite` → a derived bbs-2023 proof
verifies through the existing `IVerifier` unchanged + the holder `IBbsDeriver`); `IBbsDeriver` /
`BbsDisclosureRequest` (public, draft-free) over the internal `Bbs2023Deriver` (sole caller of
`Bbs2023Cryptosuite.DeriveProof`, F9 CSPRNG presentation header, F5 offloaded derivation, draft type
confined). FR-031 (derive, unlinkable, mandatory-always-revealed) + FR-042 (verify) satisfied. **FR-014
(issuance) is gated** — DataProofs exposes no key-store/ISigner BBS create API, so a bbs-2023 issuance request
fails fast (tested); it ships when that upstream API lands.

**Deviations from the plan (deliberate, documented):**
1. **No `InternalsVisibleTo` needed** — `IRandomSource` and `Credential.Parse` are public, so the deriver
   uses no Core internals (cleaner than §3.4 anticipated; truly zero Core changes).
2. **No Core issuance-gate code** — the substrate's own `NotSupportedException` is an accurate, contract-
   consistent gate (minimal impact; §4's "optional clean gate" skipped).
3. **`InspectBbsBase`/`BbsDisclosureMap` and `Bbs2023BaseRequest` deferred** (no public base-proof-metadata
   API; issuance gated), per §2.

**Adversarial review (2026-06-22).** Three agents (compile-and-run, ~50 exploits, BBS native lib present)
found **zero cryptographic forgeries** — mandatory enforcement, issuer binding, splice/replay resistance,
F7, NFR-005/NFR-002 all held. Addressed:
- **Medium — `DeriveAsync` malformed-pointer leak (fixed):** a bad RFC 6901 `RevealPointers` entry threw a
  raw substrate `ArgumentException`; now mapped to `CredentialFormatException` (+6 regression tests).
- **High — withheld validity/status residual (documented):** the *same inherent issuer-side limitation as
  M4 SD-JWT* — a holder withholds a non-mandatory `validUntil`/`credentialStatus` and the expired/revoked
  credential verifies. The verifier cannot detect an omission; the defence is issuer-side (mandatory group,
  cryptographically enforced). Documented on `IBbsDeriver`/`BbsDisclosureRequest` + a characterization test
  (withheld→Accepted residual; mandatory→Rejected mitigation). **M5 does not issue bbs-2023 bases**, so it is
  the third-party issuer's responsibility; this engine will enforce it when issuance ships.
- Documented, non-blocking: a bad derived proof can be steered `Failed → Indeterminate` via an unresolvable
  `verificationMethod` (a *general Data Integrity* property — pre-existing on the M1 embedded path, the same
  class the M4 envelope-resolver fix addressed; default `TreatIndeterminateAsFailure` keeps it `Rejected`).
  Applying the M4 tri-state resolution fix to the shared `NetDidVerificationMethodResolver` would harden the
  whole Data Integrity path uniformly — flagged as a focused follow-up (touches M1). Plus: derived-proof
  non-canonicality and a substrate large-message round-trip failure (both fail closed, upstream
  `dataproofs-dotnet`); and replay (deferred to the presentation/holder milestone).

**Out of scope (later milestones / upstream, flagged):** bbs-2023 issuance (FR-014, gated on a DataProofs
key-store BBS create API); `InspectBbsBase`/`BbsDisclosureMap` (needs public base-proof metadata); the full
`IHolder` + challenge-bound presentation (replay protection); the tri-state `NetDidVerificationMethodResolver`
hardening for the embedded Data Integrity path; the substrate large-message-set derive→verify fix.
